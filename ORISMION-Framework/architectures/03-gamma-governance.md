## §0 公理層：Γ 的兩條不可刪除公理

**Γ-0A 邊際利潤公理**
每一筆用戶交互的邊際成本必須低於邊際收入。違反此公理的系統不論技術多先進，都會在時間軸上確定性地死亡。LLM API 費用是最大且最不可預測的變動成本——必須被硬性約束。

**Γ-0B 合規優先公理**
法律要求的優先級高於商業利益和用戶體驗。合規要求與功能衝突時，功能必須讓步。免責聲明不可省略。資料刪除不可延遲超過法定期限。此公理不可商議。

### §0.1 公理層憲章回溯 `[V1.1 新增]`

**定理 0（憲章）：** 系統唯一終極目標 = maximize P(coherent_survival | t → ∞)。
Γ 架構的 7 支柱是此目標函數在商業維度的直接求解。

| 憲章 | Γ 映射 | 關係 |
|------|--------|------|
| L0 存續最大化 | 全部 7 支柱 | Γ 的存在理由 = 確保系統在商業上存活 |
| L0 三禁令 #1 | Γ-0A | 禁止為短期利益犧牲 Σ 穩定 |
| L0 三禁令 #2 | Γ-2 | 禁止無 Γ 價值的 Ω 過度消耗 |
| L0 三禁令 #3 | Γ-0B | 禁止為 Ω 精度犧牲 Γ 合規 |

**定理 0.1（憲章）：三體衝突裁決序**

```
FATAL 級別（任何架構） >>> 一切非 FATAL 決策

非 FATAL 優先序：
  Γ_compliance (合規) > Σ_stability (穩定) > Γ_revenue (商業) > Ω_precision (認知)
```

**Γ 對此裁決序的解讀：**
- **Γ-0B 合規 > 一切非 FATAL：** 合規罰款在 t=0 終結系統，故合規永遠最高
- **Σ 穩定 > Γ 商業：** 技術債在 t=短期 崩潰系統；營收失敗在 t=中期 耗盡資金。寧可慢點賺錢，不可架構崩塌
- **Γ 商業 > Ω 認知：** 認知精度降低在 t=長期 侵蝕信譽，但系統必須先活下來

## Γ-1 單位經濟學 (Unit Economics)

**核心定義：**
每筆 AI 分析的邊際成本 < 邊際收入。LLM API 的不可預測性使得必須在每次呼叫級別追蹤成本。

| 編號 | 規則 | 違反後果 |
|------|------|---------|
| Γ-1-F1 | 每次 LLM 呼叫 **必須** 記錄 token 消耗與成本 | 成本不可追蹤 |
| Γ-1-F2 | 定價覆蓋 **必須** 基於實際成本數據 | 定價脫離成本 |
| Γ-1-F3 | 成本超過定價 X% 時觸發警報 | 靜默虧損 |

**Σ 實現：** Σ-6.1 `llm_logs` 表（model/tokens/latency/cost/prompt_version）。

## Γ-2 Token 預算制 (Token Budget System)

**核心定義：**
Token 是不可回收的熵增（憲章 L0 三禁令 #2）。每個用戶階層有硬性 Token 上限。

| 編號 | 規則 | 違反後果 |
|------|------|---------|
| Γ-2-F1 | 每次 LLM 呼叫 **必須** 設定 `max_tokens` | Token 成本失控 |
| Γ-2-F2 | `max_tokens` 依用戶階層差異化 | 免費用戶消耗付費資源 |
| Γ-2-F3 | 預算耗盡 → Ω 降級模式（維度縮減/Phase 截斷） | 仍全力推理=虧損 |

**Ω 約束（硬性）：** Token 預算耗盡時，Ω **必須**降級推論深度，**禁止**突破上限。

## Γ-3 存取階層 FSM (Access Tier FSM)

**核心定義：**
用戶分為離散等級，每級有確定的功能與配額。等級判定由 `getAccessTier()` 唯一負責。

> **命名區分 [V1.1 補丁]：** `getAccessTier()` 返回計費階層（`free | pro | enterprise`），與 Σ Reference 中的 `getAccessLevel()` 返回功能閘門（`full | degraded | locked`）正交互補。兩者共存於同一 codebase，語義不重疊。

| 編號 | 規則 | 違反後果 |
|------|------|---------|
| Γ-3-F1 | Paywall **僅** 讀取 `subscriptions` SSOT | 多源分叉 |
| Γ-3-F2 | `getAccessTier()` 必須 exhaustive switch | 遺漏等級=權限漏洞 |
| Γ-3-F3 | 新增等級 **必須** 更新所有 switch | 新等級無配額 |

```typescript
type AccessTier = 'free' | 'pro' | 'enterprise';
function getAccessTier(subscription: Subscription | null): AccessTier {
  if (!subscription) return 'free';
  switch (subscription.status) {
    case 'active':
    case 'trialing': return subscription.plan === 'enterprise' ? 'enterprise' : 'pro';
    case 'past_due': return 'pro'; // 寬限期
    case 'canceled':
    case 'unpaid': return 'free';
    default: return subscription.status satisfies never; // exhaustive
  }
}
```

## Γ-6 合規即代碼 (Compliance as Code)

**核心定義：**
GDPR/CCPA 不是文件，是代碼。違規罰款 = 存在性威脅（憲章定理 0.1：合規 > 一切）。

| 編號 | 規則 | 違反後果 |
|------|------|---------|
| Γ-6-F1 | 帳號刪除 **必須** 清除所有個人可識別資訊（PII）。非財務表採用 `ON DELETE CASCADE`；財務相關表採用 `ON DELETE RESTRICT` + 應用層去識別化 `[V1.2 修訂]` | GDPR 罰款 / 會計紀錄消失 |
| Γ-6-F2 | 刪除在法定期限內完成 | CCPA 違規 |
| Γ-6-F3 | 個人識別資訊 PII 分類並標注 | 無法執行刪除 |
| Γ-6-F4 | LLM 日誌 **禁止** 記錄用戶原始輸入 | GDPR Art.5 |
| Γ-6-F5 | 資料處理目的 **必須** 在隱私政策明確列舉 | GDPR Art.13 |
| Γ-6-F6 | 財務相關表（incomes, expenses, invoices, audit_logs, subscriptions, payment_provider_links）外鍵策略 **必須** 為 `ON DELETE RESTRICT` `[V1.2 新增]` | 財務數據隨帳號消失 = 稽核斷裂 |
| Γ-6-F7 | 帳號註銷流程 **必須** 先執行財務數據去識別化（anonymize PII、保留金額與狀態），再刪除 auth.users `[V1.2 新增]` | 刪除順序錯誤 = RESTRICT 阻擋 |

**Σ 實現：** Σ-4.2 RLS + 非財務表 CASCADE / 財務表 RESTRICT + Σ-6.1-F3 禁記錄輸入。

### §6.5 Γ-6-F1 修訂推導鏈 `[V1.2 新增]`

**衝突源：** Γ-0B 合規優先公理有兩個互相張力的子需求——

1. **GDPR 刪除權：** 用戶有權要求刪除其個人資料 → 傾向 CASCADE
2. **會計法規稽核要求：** 財務紀錄必須可追溯 → 傾向 RESTRICT

**裁決（ADR-025 / ADR-025）：** 兩者皆正確，適用範圍不同。

| 資料類別 | 外鍵策略 | 理由 |
|---------|---------|------|
| profiles、user preferences 等個人偏好 | `ON DELETE CASCADE` | GDPR 刪除權——用戶離開時清除個人數據 |
| incomes、expenses、invoices、audit_logs | `ON DELETE RESTRICT` | 會計稽核——財務紀錄不可隨帳號刪除而消失 |
| subscriptions、payment_provider_links | `ON DELETE RESTRICT` | 金流紀錄——同上 |

**帳號註銷執行序（Γ-6-F7 的操作化）：**

```
Step 1: 財務數據去識別化
  ├─ incomes:  client_name → NULL, client_contact → NULL（保留 amount_cents, status）
  ├─ expenses: payee → NULL（保留 amount_cents, status）
  ├─ invoices: tax_id → NULL（保留 invoice_number, amount_cents）
  └─ audit_logs: metadata.snapshot 中的 PII 欄位 → NULL

Step 2: 非財務數據 CASCADE 刪除
  └─ DELETE FROM auth.users WHERE id = ? → CASCADE 清除 profiles 等

Step 3: 驗證
  └─ 確認 auth.users 已刪除 + 財務表仍存在（匿名化狀態）
```

> **設計理由：** 去識別化後的財務紀錄既滿足 GDPR（個人數據已清除）又滿足會計法規（財務紀錄匿名保留）。這是 Γ-0B 兩個子需求的最小公倍數解。

## Γ-7 AI 輸出責任 (AI Output Liability)

**核心定義：**
AI 生成內容 ≠ 專業建議。免責聲明是法律防護的最小必要條件。

| 編號 | 規則 | 違反後果 |
|------|------|---------|
| Γ-7-F1 | AI 輸出 **必須** 附帶免責聲明 | 用戶索賠 |
| Γ-7-F2 | 免責聲明 **不可** 移除/隱藏/最小化 | 法律防護失效 |
| Γ-7-F3 | 首次使用前收集 AI 免責同意 | 缺少用戶知情同意 |
| Γ-7-F4 | **禁止** 命令語氣（「你應該…」→「分析顯示…」） | 隱含建議責任 |

**免責聲明呈現矩陣：**

| 場景 | 方式 | 可省略？ |
|------|------|---------|
| 分析結果頁面底部 | React 組件（永久顯示） | **否** |
| 首次使用前 | 同意彈窗（記錄至 user_consents） | **否** |
| 服務條款 | 專門章節 | **否** |
| PDF 匯出 | 頁尾浮水印 + 首頁聲明 | **否** |

**Ω 約束：**
- Ω Prompt 最後一段 **必須** 含免責聲明模板
- Ω 置信度 < [II 類] → 追加「結果不確定性較高」提示
- Ω HALT FATAL → 直接拒絕輸出（非追加免責）

### §7.5 B2B 授權場景免責延伸 `[V1.2 新增 — ADR-012 / ADR-012]`

Γ-7 原有 F1-F4 設計對象是「系統 → 終端用戶」的直接關係。當系統以 SaaS 授權方式提供給第三方（如 B2B 客戶），多了一層責任鏈：「IP 擁有者 → 被授權方 → 終端使用者」。

**三層免責防火牆：**

| 層級 | 防火牆機制 | 保護目標 | 公理依據 |
|------|----------|---------|---------|
| L1 合約層 | 正式 SaaS 授權協議（IP 擁有者 ↔ 被授權方） | IP 資產與被授權方資產在法律上不可混同 | Γ-0B 合規優先 |
| L2 技術層 | 系統強制寫入 `liability_scope: "licensee"` 於每筆 AI 輸出 | 技術證據證明責任邊界 | Σ-6.1 可觀測性 |
| L3 輸出層 | Γ-7-F1 免責聲明擴展為雙重聲明 | 終端使用者知情同意 | Γ-7-F1 + Γ-7-F3 |

**L1 合約層強制條款：**

1. **責任上限條款：** 系統授權方對被授權方因 AI 輸出所致損失的賠償上限 = 過去 12 個月授權費總額（Γ-0A 的法律表達：責任不可超過收入）
2. **用途限制條款：** 明確列舉被授權方可將 AI 輸出用於的場景，排除「最終決策依據」（Γ-7-F4 的合約級擴展）
3. **免責傳遞義務：** 被授權方在向下游呈現 AI 輸出時，有義務保留或等效傳遞免責聲明。違反 = 被授權方自行承擔全部責任

**L3 輸出層雙重聲明模板：**

```
[系統免責] 本分析由 ORISMION AI 引擎生成，不構成專業建議。
[授權免責] 本系統由 [被授權方名稱] 作為授權用戶使用，
           系統供應方不對被授權方的使用方式承擔責任。
```

**適用條件：** 此延伸僅在 B2B SaaS 授權場景下啟用。直接面向終端用戶（B2C）的場景仍適用 Γ-7 F1-F4 原有規則。

### §9.5 Γ 在碎形架構中的位置 `[V1.1 新增 — 憲章 L4-B]`

**憲章 L4-B：** 每個 API Route 呈現碎形洋蔥結構。

**實作層序：** Γ 的 per-userId 治理（Γ-5-F2）需要已認證身份，因此 Σ Auth 是碎形的最外層前置條件：

```
請求進入
 ↓
[Σ-4.1] Auth：身份驗證（per-user Γ 檢查的前置條件）
 ↓ 通過
[Γ-5] 速率限制：配額足夠？ → 否 → 429 Too Many Requests
 ↓ 是
[Γ-3] 階層配額：此功能對此階層開放？ → 否 → 403 Upgrade Required
 ↓ 是
[Γ-2] Token 預算：今日剩餘 token？ → 否 → 429 Budget Exhausted
 ↓ 是
 === 進入 Σ 驗證層 ===
[Σ-1.3] Zod → [Σ-4.2] RLS → ...
 === 進入 Ω 層 ===
[Ω] 推理 → 輸出
 === 返回 Γ 層 ===
[Γ-1] 記帳：更新 token 消耗
[Γ-7] 免責：附帶免責聲明
```

**設計理由：** Auth 優先是因為 Γ-5-F2 強制要求 per-userId 限速（非 per-IP），userId 必須由 Σ Auth 提供。Γ 的商業過濾在認證之後立即攔截不經濟的請求，仍先於 Σ 驗證和 Ω 推理，避免消耗寶貴的計算資源。
